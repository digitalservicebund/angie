# Frontend-Entwicklung

## Utility-first Ansatz

Alle unsere Web-Projekte setzen beim CSS auf den gut skalierenden
utility-first-Ansatz. Utility-first bedeutet, primär kleine CSS-Utility-Klassen
zu komplexeren Einheiten zu kombinieren. Der Vorrat an Utilities ist quasi fix
und ist idealerweise schon am Anfang vorgegeben. Das hat einige Vorteile. In
einem traditionellen Projekt wird eigener CSS-Code geschrieben, und zwar in der
Regel mit wachsender Tendenz. Das Hinzufügen einer Komponente bedeutet dann auch
ein Anwachsen der CSS-Code-Menge. Häufig wird der CSS-Code auch nicht gelöscht,
wenn eine Komponente wieder entfernt wird. Es sammelt sich “dead code” an, der
trotzdem mit an den Browser ausgeliefert wird. Außerdem, wie der CSS-Code
strukturiert wird, wie CSS-Klassen benannt werden, welches “System” dahinter
steckt, das wird häufig in jedem Projekt neu erfunden. Wenn mehrere Menschen an
dem Code arbeiten, kommt es dann fast zwangsläufig zu immer mehr Inkonsistenzen,
da das System in der Regel auch nicht ausreichend dokumentiert wird.

## Tailwind CSS Framework

Der utility-first-Ansatz ist nicht neu, hat aber erst durch Tailwind breite
Vorbereitung gefunden. Denn entscheidend ist auch die Umsetzung von
“utility-first”. Eine einfache Liste von Utilities, die in das Projekt
eingebunden wird, ist nicht ausreichend; sie wäre schlicht zu lang. Tailwind
bietet eine riesige Auswahl an Utilities und gleichzeitig einen Mechanismus, der
sicherstellt, dass nur die tatsächlich genutzten Utilities an den Browser
ausgeliefert werden. Tailwind ist zu einem festen Bestandteil in unseren
Web-Projekten geworden. Es sorgt für Code-Konsistenz sowohl innerhalb eines
Projekts, als auch über Projektgrenzen hinweg. Sollte ich als Entwickler:in
zwischen Projekten wechseln, fühlt sich der CSS-Code immer gleich vertraut an.
Zur Homepage von Tailwind:\
[https://tailwindcss.com/](https://tailwindcss.com/)

## Tailwind Preset

Das Tailwind Framework kann umfangreich konfiguriert werden. Das kann sowohl im
nutzenden Projekt selbst, als auch zusätzlich über sogenannte Presets passieren.
Ein Preset ist eine Konfiguration, die in einem Projekt importiert werden kann
und Defaults setzt. Das Tailwind Preset von Angie ist die single source of truth
für u.a. Farben, Fonts und Spacings. Es bringt Angies Design Tokens ins Projekt
und überschreibt die sonst durch Tailwind selbst eingebrachten Defaults. Es ist
eines der durch Angie bereitgestellten Artefakte und kann als npm-Paket
installiert werden. Mehr Informationen zum Tailwind Preset:\
[https://digitalservicebund.github.io/style-dictionary/docs/how-to/use-with-tailwind/](https://digitalservicebund.github.io/style-dictionary/docs/how-to/use-with-tailwind)

## Tailwind Plugin

Das Tailwind Framework kann außerdem erweitert werden. Es bietet einen
Plugin-Mechanismus, über den wir den CSS-Code der Angie-Komponenten in das
Framework integrieren können. Wie das Preset, wird auch das Plugin als npm-Paket
installiert. Dadurch werden im Projekt zusätzliche Klassen verfügbar, um zum
Beispiel einen Button zu stylen. Die Distribution als Plugin hat gegenüber der
Bereitstellung eines separaten Stylesheets den Vorteil, dass die zusätzlichen
Klassen dem gleichen Mechanismus zur Vermeidung ungenutzten Codes unterliegen.
Mehr Infos zur Installation des Plugins:\
[https://github.com/digitalservicebund/angie#README](https://github.com/digitalservicebund/angie#README)

## JavaScript-Komponenten

Wie oben beschrieben, beschränken sich Angies Entwicklungs-Artefakte (zumindest
zurzeit) auf Tailwind, also auf CSS. Das deckt vor allem das Aussehen und nur
zum Teil das Verhalten von Komponenten ab. Unsere Projekte nutzen in der Regel
Bibliotheken oder Frameworks zur Erstellung der Benutzeroberfläche. Das könnten
zum Beispiel React oder Vue sein. Diese haben jeweils ihre eigenen Standards,
wie Komponenten (aus denen die Benutzeroberfläche zusammengesetzt ist) aufgebaut
sind. Zurzeit setzen wir gleichermaßen React und Vue in unseren Projekten ein.
Bisher lässt sich nicht absehen, in welche Richtung wir uns als DigitalService
weiterentwickeln. Daher ist es gerade nicht angezeigt, die Angie-Komponenten als
React- oder Vue-Komponenten bereitzustellen.

### Web Components

Wie eben beschrieben, bringen JavaScript-Bibliotheken/Frameworks ihre eigenen,
inkompatiblen Komponenten-Standards mit. Ein möglicher Weg, dies aufzulösen,
heißt Web Components. Genauer gesagt geht es um Custom Elements, also die
Möglichkeit, sozusagen die im Browser schon vorhandene Liste an HTML-Elementen
durch eigene Elemente zu erweitern. Das Besondere ist, dass diese Elemente sich
wie andere HTML-Elemente verhalten, insbesondere auch gegenüber
JavaScript-Frameworks. Dadurch wird ihnen eine Kompatibilität mit allen
JavaScript-Frameworks nachgesagt. Es ist geplant, Möglichkeiten in diesem
Bereich zu evaluieren.
